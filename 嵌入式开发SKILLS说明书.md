# embedded-dev Skill 说明书

by [DuncanY](https://github.com/DunCanYounG-1)

## 1. 这是什么

`embedded-dev` 是一个面向嵌入式固件开发的结构化技能包，核心是 `SKILL.md` 中定义的 **RIPER-5 嵌入式芯片开发协议**。它不是单纯的“写代码提示词”，而是一套把嵌入式开发过程标准化的工作流，覆盖：

- 芯片/平台识别
- 驱动与外设开发
- 引脚规划与冲突检测
- 数据手册与网表分析
- 驱动移植
- 中断、DMA、时钟、低功耗、看门狗等关键问题
- 长任务治理、多 Agent 协作、证据化交付
- Git 快照回退
- 四文件磁盘记忆

这个 skill 的目标不是“快速给一段代码”，而是让 AI 在嵌入式任务里按工程化方式工作，减少拍脑袋决策、减少上下文丢失、减少“看起来写了很多，实际上不可验证”的情况。

---

## 2. 这个 skill 主要解决什么问题

### 2.1 核心作用

它把嵌入式任务拆成一套强约束流程：

1. 先查证据，再给结论
2. 先识别芯片和库，再谈实现
3. 先查现成驱动/开源方案，再决定是否自己写
4. 先规划引脚、DMA、中断、时钟，再写代码
5. 每次修改后同步记录硬件资源、编辑记录、研究发现、计划状态
6. 长任务按轮次推进，支持 Scout / Builder / Verifier 分权
7. 每轮都要给验证标准和证据包，避免“理论上没问题”

### 2.2 适用场景

- STM32、ESP32、Arduino、RISC-V、NXP、TI MSP430、CH32、GD32、AT32、APM32 等 MCU/SoC 固件开发
- GPIO、UART、SPI、I2C、DMA、TIM、ADC、PWM、CAN、USB、RTOS 等外设开发
- 中断、定时器、DMA、时钟树、优先级、看门狗、低功耗相关问题
- 驱动移植、竞赛项目开发、寄存器排障
- 查数据手册、查网表、做引脚规划
- 多文件、多轮次、需要交接的嵌入式任务

### 2.3 不适用场景

- Web、移动端、桌面端等纯软件项目
- 普通通用 C/C++ 项目但不涉及硬件
- 只是写文档、做项目管理、做知识库整理
- 主要任务不是固件/硬件分析时

如果任务主线不是嵌入式开发，这个 skill 会明确要求把主执行权交给其他更合适的 skill。

## 3. 这个 skill 的核心设计思想

`embedded-dev` 的核心不是“生成代码”，而是以下几条工程原则：

### 3.1 证据优先

没有以下任一证据时，不允许宣称“已修好”：

- 代码位置
- 编译输出
- 测试结果
- 串口日志
- 数据手册依据
- 网表依据
- 实测波形/寄存器状态

### 3.2 复用优先

协议明确要求：

- 优先查本地离线索引
- 再查官方文档/Context7
- 再查开源驱动
- 最后才考虑自己实现

也就是说，这个 skill 天然偏向“找成熟方案并移植”，而不是重新造轮子。

### 3.3 长任务可恢复

它通过四个固定文件做“磁盘工作记忆”，使任务中断、上下文被压缩或多轮协作时还能恢复现场。

### 3.4 主协议精简，细则外置

主文件 `SKILL.md` 只保留最重要的总规则，大量专项流程放在 `refs/` 和 `modes/` 里按需加载，避免主协议膨胀成无法维护的“大杂烩”。

## 5. 主工作流：RIPER-5 五阶段

`embedded-dev` 的主流程分为 5 个模式，且每次回复要求在开头声明：

```text
[MODE: RESEARCH]
```

默认从 `RESEARCH` 开始，除非用户请求已经明确定位到某一阶段。

### 5.1 RESEARCH

用途：收集事实，理解项目和硬件约束。

主要工作：

- 搜索现有驱动、SDK、官方示例、开源方案
- 识别芯片平台和固件库
- 分析相关代码、ISR、时钟和外设配置
- 做引脚规划与冲突检测
- 生成或更新 `硬件资源表.md`
- 记录搜索到的候选方案到 `研究发现.md`

禁止事项：

- 不允许直接改代码
- 不允许下最终方案结论

输出要求：

- 已确认事实
- 证据来源
- 未确认问题

### 5.2 INNOVATE

用途：讨论候选方案，而不是直接选一个执行。

主要工作：

- 评估 RESEARCH 找到的开源方案
- 比较中断/轮询/DMA、自研/移植等实现路径
- 评估 CPU 负载、资源占用、兼容性

禁止事项：

- 不允许输出具体实施计划
- 不允许开始编码

### 5.3 PLAN

用途：把方案落成具体实施清单。

必须包含：

- 文件路径
- 模块边界
- API 设计
- 寄存器/外设配置要点
- 验证标准
- `review:true/false`

硬约束：

- `main.c` 只能做启动编排和主循环调度
- 不能把寄存器配置、协议解析、算法状态机大量堆进 `main()`

### 5.4 EXECUTE

用途：严格按计划实施。

规则重点：

- 一轮只推进一个改动点
- 每轮先声明 `trace_id`、目标、验证标准、停止条件
- 发现偏差必须报告
- `review:true` 步骤必须展示代码和证据，再等用户确认
- 每步确认后走 Git 自动快照

### 5.5 REVIEW

用途：验证结果是否真正符合需求和计划。

审查分三层：

1. 验证门：先跑编译/测试/实测，再允许宣称完成
2. 硬件合规：代码与 `硬件资源表.md` 是否一致
3. 代码质量：模块边界、`main.c`、`volatile`、临界区、库风格是否合规

协议明确禁止用“应该”“理论上”“大概”“基本上”来声明完成。

## 6. 长任务治理：Vibe 执行壳

这个 skill 融合了 `refs/vibe-workflow.md` 的长任务治理规则，适用于：

- 修改 2 个及以上文件
- 预计要多轮编译/烧录/调试
- 需要多 Agent 并行
- 需要回退和交接

### 6.1 三角色分工

- `Scout`：只收集证据，不写代码
- `Builder`：只做最小实现
- `Verifier`：只做审查和验收

### 6.2 轮次制

每轮必须有：

- `trace_id`
- 当前轮次
- 目标
- 验证标准
- 停止条件
- 证据包

### 6.3 红线

出现以下情况要立即停止并回 PLAN 或请用户裁决：

- 新增依赖
- 改动超过 5 个文件
- 修改根目录构建/烧录/工具链配置
- 连续两轮无法确认根因
- 需要危险操作

## 7. 四文件磁盘工作记忆

这是这个 skill 最有特色的机制之一。它要求在工程根目录维护以下四个文件：

- `项目规划清单.md`
- `编辑清单.md`
- `硬件资源表.md`
- `研究发现.md`

### 7.1 各文件作用

#### `项目规划清单.md`

保存：

- 当前 RIPER 阶段
- trace_id
- 当前轮次
- 下次继续点
- 停止条件
- 功能模块状态
- 轮次计划

#### `编辑清单.md`

保存：

- 每次代码编辑的文件与内容
- 验证标准和结果
- Git 提交 hash
- 是否已 push
- 是否有回档
- 长任务时的角色和轮次

#### `硬件资源表.md`

保存：

- 芯片型号
- 开发框架
- 引脚分配
- DMA 通道分配
- 中断优先级矩阵
- 时钟配置
- 变更记录

这份文件是全流程共享的“硬件事实基线”。

#### `研究发现.md`

保存：

- 搜索关键词
- 工具来源
- 技术发现摘要
- 可信度
- 已采纳/已排除状态
- API 用法记录
- 排查线索
- 三次失败记录

### 7.2 五问重启测试

每次新会话或上下文恢复时，协议要求先检查四个文件并回答五个问题：

1. 当前阶段是什么
2. 最近改了什么
3. 当前硬件资源情况如何
4. 之前研究发现了什么
5. 现在该以哪个角色继续

### 7.3 使用价值

这套机制解决了嵌入式 AI 协作里最常见的三个问题：

- 上下文断了以后不知道做到哪一步
- 改过什么记不清
- 硬件资源约束反复漂移

## 8. Git 快照与回档机制

协议要求在 EXECUTE 阶段每完成一个实施项并得到用户确认后，做**本地**存档（**绝不自动 push**，**不用 `git add -A`**，按 PLAN 清单显式 `git add <file>...`）：

```bash
git add <本步骤涉及的具体文件>
git commit -m "[AUTO-SNAPSHOT] 步骤N: <任务摘要>"
# 不自动 push；推送由用户显式发起
```

> 完整硬规则（敏感文件预检、不用 `git add -A`、绝不自动 push）见 `refs/git-snapshot.md` 与 `SKILL.md` Git 备份与回档规则。

回档规则：

- 用户说“回档/回退/退回上一步”时，优先走 Git 回退流程
- 默认使用保守方式：`git revert --no-edit HEAD`
- 回退前先 `git stash`
- 只有用户明确接受丢失改动时才允许 `git reset --hard HEAD~1`

这样做的好处是：回退是“真回退”，而不是手工瞎改回去。

## 9. 工具体系与调用优先级

这个 skill 为嵌入式开发建立了一套明确的工具路由。

### 9.1 总体优先级

1. 本地 refs
2. Context7 / grok-search / `gh` / Document Skills 等工具
3. 手动分析
4. 最后才依赖经验猜测

### 9.2 关键工具说明

#### Grok-Search

主联网检索工具，优先用于：

- 查驱动
- 查报错
- 查竞赛经验
- 查数据手册入口
- 查版本信息

#### Context7

主要用于查库文档和 API：

- STM32 HAL
- STM32 StdPeriph
- ESP-IDF
- Arduino
- FreeRTOS
- CMSIS

#### Document Skills

主要用于读文档：

- `/pdf`：数据手册、参考手册
- `/xlsx`：引脚表、BOM、测试数据
- `/docx`：规格说明文档
- `/pptx`：答辩资料

#### Sequential Thinking

用于复杂推理，不用于替代事实检索。常见场景：

- 引脚冲突分析
- DMA 分配
- 中断优先级设计
- 复杂故障排查

#### `gh` CLI

用于 GitHub 搜索开源驱动、读取仓库内容、克隆候选项目。

#### Embedded Debugger / Serial MCP

仅在连接开发板后用于在线烧录、读寄存器、下断点、串口交互。

### 9.3 工具降级策略

如果主工具不可用，协议明确要求有降级方案。例如：

- Context7 不可用 → 本地 refs → 官方 PDF → grok-search
- grok-search 不可用 → WebSearch → 用户手动搜索
- Document Skills 不可用 → 内置 Read

## 10. 扩展模式（modes）

`modes/` 目录放的是按需调用的专项流程。

### 10.1 `competition.md`

比赛模式，触发词：`启用比赛模式`

适用：

- 电赛
- 蓝桥杯嵌入式
- 竞速开发、多人分工

特点（v2 当前版本）：

- 定义 6 个独立 subagent：`[ARCH] / [MATLAB] / [DRV] / [ALG] / [QA] / [REPORT]`
- 强制 Git 阶段检查点 `CP-0` 到 `CP-5`（含 CP-0a/0b、CP-1.5 仿真门）
- CP-1 先冻结引脚、DMA、中断优先级和接口契约
- CP-1.5 由 MATLAB 做仿真验证（控制/算法题）
- CP-2 允许 DRV 与 ALG（+REPORT）并行开发
- CP-3 QA 以嵌入式专项清单做强审查（含 MIL/SIL/PIL）
- CP-4 由 ARCH 集成主循环、调试宏、时间片任务，CP-5 答辩演练
- 强调“故障即安全”“看门狗必启”“零 TODO”“零裸轮询”
- 视觉题由独立 `auto-vision` skill 承担

这是一个替代型模式，会替换标准 RIPER-5 的阶段流程，但基础规范仍生效。

### 10.2 `datasheet-lookup.md`

数据手册查阅模式，触发词：`查手册` / `查数据手册` / `datasheet`

用途：

- 搜索并下载 PDF
- 提取引脚、寄存器、时序、电气参数
- 把“魔数”写回代码注释并标明来源页码

流程：

1. 定位需要的文档类型
2. 获取 PDF
3. 用 `/pdf` 定向读取
4. 结构化提取参数
5. 用 `@ref` 注释写回代码

### 10.3 `netlist-lookup.md`

网表读取模式，触发词：`网表` / `netlist`

用途：

- 自动搜索网表文件
- 识别 Protel/KiCad 等格式
- 提取 MCU 引脚连接
- 生成引脚映射表
- 与 `硬件资源表.md` 比对
- 生成代码中的引脚定义和 `@netlist` 注释

核心原则是：**有网表时，网表优先于人工猜测。**

### 10.4 `seekfree-lib.md`

逐飞库管理模式，触发词：`逐飞` / `seekfree`

用途：

- 英飞凌或竞赛类芯片优先复用逐飞科技开源库
- 管理本地 `逐飞库索引.md`
- 查本地缓存，没有再下载
- 结合主协议做驱动移植

### 10.5 `mcp-healthcheck.md`

MCP 工具健康检查模式，触发词：`检查工具` / `mcp检查` / `healthcheck`

用途：

- 检查当前 skill 真正依赖的 MCP/文档工具是否可用
- 最小化测试 Context7、Grok-Search、Document Skills、Sequential Thinking 等
- 检查配置、权限、依赖与网络问题
- 产出工具可用性报告

## 12. hooks 与本地配置

这个 skill 不只是文档，还带了一些自动提醒能力。

### 12.1 `SKILL.md` 中的 hooks

#### `UserPromptSubmit`

当工程目录里存在以下文件时自动提醒读取最新内容：

- `项目规划清单.md`
- `编辑清单.md`
- `硬件资源表.md`
- `研究发现.md`

#### `PreToolUse`

在写文件/编辑/执行 Bash 前，尝试输出：

- `项目规划清单.md` 前 20 行
- `硬件资源表.md` 的芯片与开发环境部分
- `研究发现.md` 最近发现

目的是让执行动作前先看到当前上下文。

#### `PostToolUse`

在修改代码后提醒同步更新：

- `编辑清单.md`
- `硬件资源表.md`
- `研究发现.md`

### 12.2 hooks 兼容性说明

这些 hooks 使用 POSIX shell 语法，文档中明确说明：

- 在 bash/zsh 环境下应正常工作
- 在纯 PowerShell 环境下 hooks 可能受限
- 即便 hooks 失效，核心协议仍然有效

### 12.3 `.claude/settings.local.json`

该文件给当前 skill 放开了一些本地权限和能力，例如：

- 若干 Bash 检索命令
- WebFetch / WebSearch
- Context7、xai-search、MindOS 等工具权限
- `npm`、`npx`、`python` 等必要操作

这使得 skill 在需要检查工具、装依赖、查配置时能更顺畅地工作。

## 13. 实际使用方法

### 13.1 最常规的用法

直接把嵌入式需求交给它，例如：

```text
帮我看一下 STM32F103 的 USART1 中断为什么不触发
```

它会按协议：

1. 先识别芯片和库
2. 进入 RESEARCH
3. 检查代码、外设、中断和资源表
4. 再进入 INNOVATE / PLAN / EXECUTE / REVIEW

### 13.2 查手册

```text
查手册，确认 STM32F103 ADC 时钟上限和采样时间
```

它会触发 `datasheet-lookup`，然后把参数写成可追溯的引用格式。

### 13.3 查网表

```text
查网表，确认 USART1 和 MPU6050 的引脚连接
```

它会优先找工程里的网表，提取 MCU 引脚映射，再与资源表比对。

### 13.4 驱动移植

```text
帮我给 STM32F103 移植一个 SSD1306 驱动，优先找成熟项目
```

它会先查本地索引、再查开源驱动、再评估是否移植。

### 13.5 比赛模式

```text
启用比赛模式，做一个平衡车控制系统
```

它会进入 6-agent 并行式流程，先冻结硬件资源与接口，再让驱动和算法分角色推进。

### 13.6 工具检查

```text
检查所有 mcp 工具
```

它会只检查与本 skill 相关的工具，不会泛化到无关 MCP。

## 14. 推荐的使用姿势

为了让这个 skill 发挥最大价值，建议这样使用：

1. 给出明确芯片型号、框架、IDE 和问题现象
2. 尽量提供工程目录、关键代码、网表、数据手册或报错日志
3. 多文件/长任务时接受四文件机制，不要只依赖聊天上下文
4. 允许它先做 RESEARCH，不要一开始就要求“直接给代码”
5. 硬件相关问题尽量附上：
   - 芯片型号
   - 原理图或网表
   - 当前引脚分配
   - 编译输出
   - 逻辑分析仪/串口/示波器现象

## 15. 这个 skill 的强项

- 工程化而不是一次性回答
- 非常适合带硬件约束的任务
- 能处理从资料检索到代码实现再到验证的完整链路
- 对长任务和多轮协作友好
- 适合竞赛、驱动开发、复杂排障、跨平台迁移
- 对“引脚、DMA、中断、时钟、时序、资源冲突”这些传统 AI 容易答偏的点管得很严

## 16. 一句话总结

`embedded-dev` 本质上是一个面向嵌入式开发的“工程执行协议 + 专项知识库 + 长任务治理框架”。  
它最适合用在需要真实落地、可验证、可回退、可交接的 MCU/固件任务中，而不是只追求一次性生成代码。
